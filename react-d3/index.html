What is D3.js?

  • Data-Driven Documents
  • Low level, General Purpose Visualization Library
  • Manipulates data based documents
  • Open web standards (SVG, HTML, CSS and now Canvas)
  • Allows interactions with your graphs
  • D3 stands for data-driven documents
  • Huge API and focused on mathematical transformations of data
  • Based on web standards as SVG, HTML, CSS and canvas
  • One of it’s more celebrated features is it’s seamless integration of animations and the way it allows us to create engaging interactions.
  • Front End toolbox for creating interactive visualization, but it is not a charting library. So if you are looking for a package that gives you a bunch of ready made charts, d3 by itself won’t help you, but you could choose among several projects based on d3.

How does it work?

  • Loads data
  • Binds data to elements
  • Transforms those elements
  • Transitions between states
  • D3 loads data and attaches it to the DOM.
  • Binds that data to DOM elements and transforms those elements, transitioning among states if necessary.

    Update, Enter and Exit Pattern

  • when we joined the selection with an array of data,
  • it results in three new selections
  • that represent the three possible states: enter, update, and exit.
  • Since the selection was empty, the update and exit selections are empty,
  • while the enter selection contains a placeholder for each new datum.

    • .style()

    React Fiber

    • New reconciliating algorithm
    • Renderer and Reconciler are separate
    • From batching DOM operations, now React can...
      • Prioritize tasks
      • Split into chunks and schedule
      • Parallelize operations
    A Cartoon Intro to Fiber by Lin Clark
    • Fiber is the new reconciliation algorithm of React.
    • Reconciler and Renderer are separated so that the renderer could be different (native, VR, etc)
    • React before was batching DOM changes
    • This new reconciler is better because it takes into account priorities, parallel computations and schedules to keep the 60FPS

    • Now let's see what do they have in common and the challenges they trigger

    Commonalities

    • Help us with the DOM
    • Pure functions
    • React, and D3 are two libraries that share a common trait: They want to help us deal with DOM changes in an optimized way.
    • They both work in a “stateless mode,” that is, given an input, the rendered output will always produce the same result.

    Challenges

    • D3 creates and transforms the DOM
    • React also transforms the DOM, and keeps track of it
    • Not meant to work together
    • In React, there is state, and this information specifies what is going to be shown in the UI.
    • However, on D3, the state is kept on the DOM itself, like we did on the jQuery times.
    • It could be a class on your tooltip that makes it show. The data itself is on the DOM as well.
    • They are opinionated libraries.

    Choosing a project

    Choosing criteria

    • They are a lot of variables here
    • As a good engineer, I will try to reduce some complexity

    Quality

    • Well tested
    • D3 V4
    • Great docs
    • We are going to set it

    Time

    • Long term investment?
    • Needs to be done ASAP?
    • Team skills
  • Roll your own D3, wrap with React. Libraries don't assure you 5 years of life.
  • Grab the closest library to your specs, fork it and contribute
  • Lot's of D3 devs, keep it splitted

    • Scope

    • Highly customized vs basic charts
    • One-off work vs dataviz heavy product
  • you have a UX and Designer with you? Roll your own D3, or customize the closest Library
  • Is basic, go for any library
  • is it a single scientific or periodistic piece? Unless is in a library out of the box, go with regular D3.
  • Roll your own D3 library or contribute to another.
