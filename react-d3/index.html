What is D3.js?

  • Data-Driven Documents
  • Low level, General Purpose Visualization Library
  • Manipulates data based documents
  • Open web standards (SVG, HTML, CSS and now Canvas)
  • Allows interactions with your graphs
  • D3 stands for data-driven documents
  • with a huge API
  • and focused on mathematical transformations of data
  • Based on web standards as SVG, HTML, CSS and canvas
  • One of it’s more celebrated features is it’s seamless integration of animations and the way it allows us to create engaging interactions.
  • Front End toolbox for creating interactive visualizations, and it is not a charting library.
  • We will review several projects based on d3.

How does it work?

  • Loads data
  • Binds data to elements
  • Transforms those elements
  • Transitions between states
  • D3 loads data and attaches it to the DOM.
  • Binds that data to DOM elements
  • and transforms those elements, transitioning among states if necessary.

D3 Data Joins

  • I am going to touch the Data Joins definition a bit
  • As we will establish similitudes with the way React works

    • Transforms the DOM by selecting elements and joining to data

  • It's another approach
  • With jquery we accessed elments
  • with backbone we render views
  • with react we are smart about renders, using the virtual DOM
  • with D3 we have data joins

    • Data Join

  • Imagine we have this 5 letters in the DOM
  • We select it, and add data to create the data join
  • Three groups, the ones that stay (update),
  • the ones that need to be created (enter)
  • and the ones that need to leave (exit)
  • this is represented with an object with three arrays
  • It's a diff algorithm, but instead of comparing already rendered DOM, we compare data
  • The data is in the DOM, we'll compare this with the use of keys in React components

    • Update, Enter
    and Exit Pattern

    Resources
  • we are looking at the relationship between a selection and data, and then working with them through the three enter, update and exit groups.
  • Data points joined to existing elements produce the update (inner) selection.
  • Leftover unbound data produce the enter selection (left), which represents missing elements.
  • the remaining unbound elements produce the exit selection (right), which represents elements to be removed.

    Commonalities

    • Help us with the DOM
    • Love pure functions
    • React, and D3 are two libraries that share a common trait: They want to help us deal with DOM changes in an optimized way.
    • They both work in a “stateless mode,” that is, given an input, the rendered output will always produce the same result.

    Challenges

    • D3 creates and transforms the DOM
    • React as well, and keeps track of it
    • Not meant to work together
    • React keeps track of it entirely
    • They are opinionated libraries.

    Choosing an approach

    * Sometimes we will be choosing in base on the context and history of the platform we are building.

    Choosing criteria

    * They are a lot of variables here * As a good engineer, I will try to reduce some complexity
    • Well tested
    • D3 V4
    • Great docs
    • We are going to set it
    • Long term investment?
    • Needs to be done ASAP?
  • Roll your own D3, wrap with React. Libraries don't assure you 5 years of life.
  • Grab the closest library to your specs, fork it and contribute
    • Highly customized?
    • Basic charts?
    • One-off work?
    • Dataviz heavy product?
  • you have a UX and Designer with you? Roll your own D3, or customize a Library
  • Is basic, go for the closest library for your specs
  • is it a single piece? Unless is in a library out of the box, go with regular D3.
  • Roll your own D3 library or contribute to another.
    • Budget
    • Training
  • the skills of your team

  • More React: React-D3 Libraries, Lifecycle methods with examples; don't roll your own.
